API 公開鍵認証 導入・設定マニュアルv4r9

（認証情報暗号化・自動運用設定編）

ダウンロードしたAPIプログラムをLinux環境に導入し、認証IDや秘密鍵を安全に暗号化（独自暗号化によるセキュア保存）した上で、systemd を用いて自動実行するための設定手順書です。

※ 本マニュアルの認証方式は、従来の『認証局（CA）の発行するデジタル証明書』を用いたPKI認証とは異なり、FIDO/パスキーの仕組みに準拠した公開鍵認証を採用しています。そのため、クライアント証明書のインストール作業は不要です。

動作確認環境は、API v4r9 / Python 3.13.5 / debian13 です。

固定IP指定の推奨
・秘密鍵、第2パスワードファイル、またはログインレスポンスファイルが万が一流出した場合、第三者に不正ログインされるリスクがあります。
・安全のため、接続元を固定IPに限定する設定（IP制限）を行ってのご利用を強く推奨いたします。

ご注意
・本番環境とデモ環境では、別々のAuthID、秘密鍵、公開鍵のセットになります。
・デモ環境でv4r9を利用する場合、デモ環境の標準web画面にログインしていただき、
　デモ環境専用のAuthID、秘密鍵、公開鍵のセットを取得してください。

ℹ️ 本APIの接続方式について

本APIは、プログラミング初心者や非ITエンジニアの方にも利用しやすいよう、URLにJSON形式のパラメーターを付加して送信する独自方式を採用しています。

一般的なWeb APIとは異なる構成ですが、HTTPヘッダーやPOSTデータなどの知識を最小限に抑えながら利用できることを重視しています。

そのため、本APIはURLとJSON文字列を組み立てて送信するだけで利用でき、特別な知識を必要としない接続を提供しています。

ℹ️ 公式リファレンスとの関係について

本マニュアルおよび付属のスクリプトは、公式の ｅ支店 API マニュアル に準拠した設計となっています。公式リファレンスに記載されている公開鍵認証方式を、Linuxサーバー上で安全（秘密鍵の生ファイル放置リスクを排除）に自動運用するための実践的なセットアップ手順を提供しています。

ℹ️ 本マニュアルでの接続先

本マニュアルおよびサンプルプログラムでは、誤って実際の取引を行うことを防ぐため、デモ環境（demo-）への接続を例示しています。

本番環境を利用する場合は、接続先URLをお客様ご自身で変更してください。 接続先を変更する際は、接続先が本番環境であることを十分確認のうえ実施してください。

1. ディレクトリ構成の作成と権限設定

安全な運用のために、実行用ディレクトリと鍵を格納する隠しディレクトリを作成し、所有者以外のユーザーからのアクセスを遮断（権限700）します。

推奨ディレクトリ構成

~/e_api/                        ← API実行基盤（権限: 700 / 所有者のみアクセス可）
├── .auth/                      ← 鍵・暗号化データ格納（権限: 700）
│   ├── file_pwd2.txt           ← 第2パスワード保存ファイル（手動作成）
│   └── file_login_response.txt ← ログイン応答出力先（自動生成）
├── .venv/                      ← Python仮想環境（自動生成されます）
├── e_api_encode_auth.py        ← 暗号化ユーティリティ
├── e_api_login_pubkey.py       ← ログイン実行スクリプト
└── file_url_info.txt           ← API接続情報ファイル（手動作成）

設定コマンド

ターミナルを開き、以下のコマンドを順番に実行してディレクトリを作成・保護してください。

# [1] ディレクトリの作成
$ mkdir -p ~/e_api/.auth

# [2] ディレクトリの権限（パーミッション）制限
# 親ディレクトリを含め、所有者のみに全権限を付与し、他ユーザーを完全に締め出します
$ chmod 700 ~/e_api
$ chmod 700 ~/e_api/.auth

2. Python仮想環境の準備

システムのPython環境に影響を与えず、プログラムに必要な依存ライブラリ（通信・暗号化モジュール等）を独立してインストールするため、仮想環境を作成します。

2.1. システムパッケージの確認と導入

Debian系OS（debian/Ubuntuなど）では、仮想環境の構築に必要なOS側のパッケージが標準で入っていない場合があります。環境構築をスムーズに行うため、以下の手順で事前準備を行います。

# [1] e_api ディレクトリに移動
$ cd ~/e_api

# [2] venvパッケージのインストール
$ sudo apt update
$ sudo apt install -y python3-venv

⚠️ 「ユーザー名 は sudoers ファイルにありません」とエラーが出る場合

Linuxの初期設定によっては、操作している一般ユーザーに管理者権限（sudo権限）が割り当てられておらず、コマンドが拒否されることがあります。その場合は、一度以下の手順で管理者（root）ユーザーに切り替えてからインストールを行ってください。

# 1. 一時的に管理者（root）ユーザーに切り替える（rootのパスワードを入力） $ su - パスワード: #   # 2. パッケージ名簿を最新化し、インストールを実行する # apt update # apt install -y python3-venv   # 3. インストールが終わったら、元の一般ユーザーの画面に戻る # exit logout $

💡 パッケージが見つからない、またはバージョンエラーが出る場合のヒント

もし python3-venv が見つからない、あるいは 404 Not Found などのエラー画面が表示されて進行が止まった場合は、公式サーバーのキャッシュが古くなっているか、セキュリティニュースの確認画面で一時停止しています。

その場合は、本マニュアルの後半にある「補足：パッケージのインストールでエラーになる場合の対処法」の手順（管理者権限でのキャッシュリセットおよび q キーによるニュースの離脱）に従って実行してください。

2.2. 仮想環境の作成とライブラリ導入

# 1. 仮想環境（.venv）の作成 $ python3 -m venv .venv   # 2. 仮想環境の有効化（アクティベート） $ source .venv/bin/activate   # 3. 必須パッケージのインストール # ※ プログラムで使用する通信・暗号化用の外部ライブラリを一括導入します # ※証明書（CA）を使わない代わりに、デバイス側で鍵を安全に暗号化保存（Cryptodome）するためのライブラリを導入します (venv) $ pip install --upgrade pip (venv) $ pip install urllib3 cryptography pycryptodomex   # 4. （確認）指定した3つのライブラリが正しくインストールされたか確認 (venv) $ pip list   # 5. 仮想環境から抜ける (venv) $ deactivate $

※以降の作業でターミナルを一度閉じた場合は、再度 cd ~/e_api && source .venv/bin/activate を実行して仮想環境に入ってください。

  • e_api_private_key.der

ご注意：本番環境とデモ環境では、別々のAuthID、秘密鍵、公開鍵のセットになります。本番環境のファイルセットで、デモ環境にログインできません。デモ環境のファイルセットで本番環境にはログインできません。

3.2. プログラムファイルの配置

配布された以下の2つのPythonスクリプトを、~/e_api/ ディレクトリ直下に配置してください。

  • e_api_encode_auth.py
  • e_api_login_pubkey.py

3.3. API接続情報ファイル（手動作成）

接続先や暗証番号を記録するファイルを ~/e_api/file_url_info.txt として新規作成します。例ではエディターとしてnanoを使っていますが、vi等のご自身の使い慣れたエディターで編集してください。

$ nano ~/e_api/file_url_info.txt

環境（本番/デモ）に合わせて以下の内容を記述し、保存してください。文字コードはUTF-8で保存します。

{
    "sUrl": "https://demo-kabuka.e-shiten.jp/e_api_v4r9/",
    "sJsonOfmt": "5"
}

3.4. 第2パスワード保存ファイル（手動作成）

注文、訂正、取り消しの処理で個別に利用する第2パスワードファイル（ログイン処理では未使用）を ~/e_api/.auth/file_pwd2.txt として新規作成します。

$ nano ~/e_api/.auth/file_pwd2.txt

⚠️ 記述時の重要な注意点

このファイルには第2パスワードを保存します。その際、末尾に改行コードや余分な制御文字が混入しないよう注意して記述してください。

第2パスワードは、注文、訂正、取消で使います。慎重に取り扱ってください。また短い期間で変更してお使いください。

また、ログイン後に同じディレクトリに出来るログイン応答出力（file_login_response.txt）も同様に慎重に扱ってください。

3.5. 配置ファイルの権限変更

万が一に備え、配下のすべてのファイルも所有者のみ読み書き可能（600）に制限します。

$ chmod 600 ~/e_api/.auth/*
$ chmod 600 ~/e_api/file_url_info.txt
$ chmod 600 ~/e_api/*.py

4. 初回限定：鍵情報の暗号化と原本の削除

認証IDと秘密鍵、第2暗証番号を一つにまとめ、強固に暗号化した設定ファイル（secure_config.enc）を生成します。

4.1. 暗号化プログラムの実行

$ cd ~/e_api
$ python3 e_api_encode_auth.py

4.2. 出力結果の記録（最重要）

実行が成功すると、画面に以下のようなメッセージが表示されます。

============================================================
【重要】暗号化ファイルの作成に成功しました。
作成されたファイル: /home/[実行ユーザー名]/e_api/.auth/secure_config.enc
------------------------------------------------------------
以下の文字列を、自動実行環境の「環境変数 (API_DECRYPT_KEY)」に設定してください。
この鍵はファイルとしてディスクに保存せず、環境変数としてメモリから注入します。

API_DECRYPT_KEY=...（ランダムな長い文字列）...=

上の行の内容は /etc/systemd/system/e_api_login.service ファイルに設定してください。
============================================================

⚠️ 注意

画面に表示された API_DECRYPT_KEY=〜 の文字列は、次のステップで systemd の設定に使用します。必ず一時的にメモ帳などへ控えておいてください。

4.3. 生ファイルの安全な削除

暗号化ファイル（secure_config.enc）が正常に生成されたため、サーバー上に残った暗号化されていない原本ファイルは安全のために速やかに削除します。

$ rm ~/e_api/.auth/e_api_authid.txt
$ rm ~/e_api/.auth/e_api_private_key.pem
$ rm ~/e_api/.auth/e_api_private_key.der

4.4. 本サンプルにおける認証情報保護について

本サンプルでは、認証IDや秘密鍵を平文ファイルで保存しないよう、 暗号化設定ファイル（secure_config.enc）を利用しています。

公開鍵認証自体には必須ではありませんが、認証情報の漏洩リスクを 低減するための対策として実装しています。

本サンプルで採用している方式は、認証情報を平文で保存する運用を 避けるための実装例です。利用環境や運用方針に応じて、
 ・OSの資格情報ストア
 ・Windows DPAPI
 ・Linux Secret Service
 ・TPM や YubiKeyなどのHSM（ハードウェア・セキュリティ・モジュール） の採用
など、本サンプルと同等以上の安全性を確保できる方式を採用してください。

なお、本サンプルでは、認証情報を格納した暗号化ファイル （secure_config.enc）と、復号に必要なキー（API_DECRYPT_KEY）を 分離して管理しています。

暗号化ファイルのみ、または復号キーのみが漏洩した場合には、 認証情報が直ちに利用されることがないよう配慮しています。 しかし、両方が第三者に取得された場合には認証情報が復元される 可能性があります。

そのため、secure_config.enc および API_DECRYPT_KEY は適切な アクセス権限のもとで管理し、第三者へ開示しないよう十分注意してください。

5. systemdによるサービス化（基盤設定）

OS管理機能（systemd）にプログラムを登録することで、環境変数を安全に隠蔽しながら実行できる環境を整えます。

5.1. サービス定義ファイルの配置

配布された e_api_login.service を、システムの配置ディレクトリにコピーします（要管理者権限）。

# 1. 一般ユーザー（作業ディレクトリ）の場所にいることを一度確認 $ cd ~/e_api   # 2. 一時的に管理者（root）ユーザーに切り替える（rootのパスワードを入力） # ※「su」にすることで、一般ユーザーの現在地（~/e_api）を維持したまま root になれます $ su パスワード: #   # 3. 現在地（./）にあるサービスファイルを、システム領域へコピーする # cp ./e_api_login.service /etc/systemd/system/e_api_login.service   # 4. セキュリティのため、ファイルの権限を所有者（root）のみ（600）に制限する # chmod 600 /etc/systemd/system/e_api_login.service   # 5. コピーと権限設定が成功したか（ -rw------- になっているか）確認 # ls -l /etc/systemd/system/e_api_login.service -rw------- 1 root root 597 6月 3 16:02 /etc/systemd/system/e_api_login.service

5.2. サービスファイルの編集

コピーしたファイルを編集し、実行ユーザー名や環境変数を各自の環境に書き換えます。例ではエディターnanoを使っていますが、vi等のご自身の使い慣れたエディターで編集してください。

# nano /etc/systemd/system/e_api_login.service

以下の項目を、お客様の環境に合わせて修正してください。

  • User= : 実行するLinuxのユーザー名（例: [実行ユーザー名]）
  • WorkingDirectory= : ディレクトリのフルパス（例: /home/[実行ユーザー名]/e_api/）
  • Environment= : 手順4の4.2.でメモした API_DECRYPT_KEY の値を貼り付けます。
  • ExecStart= : 仮想環境内のPythonパスと実行スクリプトのフルパス

【設定例】

[Unit]
Description=My Python Crypto API Service
After=network.target

[Service]
Type=simple
User=[実行ユーザー名]
WorkingDirectory=/home/[実行ユーザー名]/e_api/

# 手順4で生成されたキーをここに設定
Environment="API_DECRYPT_KEY=...（あなたのキー）..."

# 必ず仮想環境(.venv)下のpython3を指定してください
ExecStart=/home/[実行ユーザー名]/e_api/.venv/bin/python3 /home/[実行ユーザー名]/e_api/e_api_login_pubkey.py

[Install]
WantedBy=multi-user.target

5.3. systemdの設定反映

設定ファイルを保存したら、以下のコマンドでシステムに設定をリロードさせます。

# systemctl daemon-reload

6. 運用・実行方法と障害対応

セットアップ完了後の、手動での実行方法、障害発生時のセーフティ仕様、およびタイマー（定時自動実行）の設定手順です。

6.1. 手動実行と状態確認の方法

プログラムをテスト起動したい場合、また実行したプログラムの処理状態や成否ログを確認したい場合は、以下のシステム管理コマンドを利用します。

■ サービスの手動起動（実行リクエスト）

# systemctl start e_api_login.service

■ サービスの実行ステータス・ログ確認

# systemctl status e_api_login.service

※ログにエラーが出ておらず、隠しディレクトリ内に ~/e_api/.auth/file_login_response.txt が正しく生成・復号されていれば、プログラムは正常に動作しています。

⚠️ ログ確認時の注意：画面が止まって元の入力画面に戻れなくなった場合

エラー原因を調べるために systemctl status を実行した際、画面の最下部に : や (END) と表示され、コマンドを入力できなくなることがあります。

これはフリーズではなく、システムがログの閲覧モード（スクロール可能な状態）を開いているためです。確認が終わったら、キーボードの q キー（Quit）を1回押してください。閲覧モードが終了し、安全に元のコマンドプロンプト（# や $）に戻ることができます。

🔍 サービスが起動しない（Failedになった）ときのトラブル確認リスト

サービスを起動した際にエラーが出た場合は、焦らずに以下の4つのチェックポイントを上から順番に確認してください。ほとんどのエラーはこれで解決できます。

【チェック 1】ログに「status=217/USER」と表示されていませんか？

原因： サービスファイル（.service）内に書かれている実行ユーザー名（User=）が、システムに存在しない名前（またはプレースホルダーのまま）になっています。

対策： ファイルをエディタで開き、[Service] セクションの User=[実行ユーザー名]、Group=user の箇所が、実際のLinux一般ユーザー名と完全に一致しているか確認・修正してください。修正後は必ず systemctl daemon-reload を実行してから再起動します。

【チェック 2】ファイル末尾に「^M」という謎の文字が付いていませんか？

原因： Windows上のメモ帳などでサービスファイルを作成・編集してLinuxに転送した場合、Windows特有の改行コード（CR）がゴミ（^M）として混入し、システムがファイルを正常に読み込めなくなります。

対策： 管理者権限で sed -i 's/\r//' /etc/systemd/system/e_api_login.service を実行し、改行コードを一括除去してください。

【チェック 3】ファイルの権限（パーミッション）は適切ですか？

原因： システムを保護するため、サービスファイルは管理者（root）以外が閲覧・改ざんできないように設定する必要があります。

対策： chmod 600 /etc/systemd/system/e_api_login.service を実行し、権限を絞ってください。ls -l を実行した際、先頭が -rw------- になっていれば正常です。

【💡 共通注意】状態確認（status）から元の画面に戻れなくなったら？

systemctl status を実行した後に画面最下部に : や (END) が出てフリーズしたように見えるのは、ログ閲覧モードが開いているためです。キーボードの q キーを1回押せば、いつでも安全に元のプロンプト画面に戻れます。

6.2. 通信タイムアウトと自動リトライ（運用上の重要仕様）

API側のサーバーは、夜間や週末（土日祝日）のメンテナンス時間帯に通信が遮断されるケースがあります。本プログラムは運用時のハングアップを防ぐため、以下の通信セーフティロジックが自動作動します。

  • タイムアウト制限: サーバーからの応答が15秒間ない場合、通信を自動で切り上げます。
  • 自動リトライ: 通信失敗を検知すると、5秒間の待機を挟んで最大3回まで自動的に再接続を試みます。
  • エラーハンドリング: すべてのリトライが失敗した場合は、きれいなエラーメッセージを標準エラー出力へ吐き出し、ステータス「1 (FAILURE)」で安全にプロセスを終了します。

詳細な生ログ（直近150行分）を直接確認したい場合は、一般ユーザー画面から以下のコマンドを実行してください。

# サービス専用の出力ログを確認する（画面が止まらずにプロンプトに戻るオプション付き） $ journalctl -u e_api_login.service --no-pager -n 150

6.3. 指定時刻に自動実行する設定（タイマー設定）

毎日決まった時刻（例：毎朝 08:30）にAPIログインを自動で実行させるため、systemdのタイマー機能（.timer）を作成します。ここからの作業はすべて管理者権限（root）で行います。

1. 一時的に管理者（root）ユーザーに切り替えます。
※suにすることで、現在の作業ディレクトリ（~/e_api）を維持したままrootになれます。rootのパスワードを入力してください。

$ su パスワード: #

2. 【手順 A】タイマー定義ファイルをシステム領域に新規作成します。

💡 補足：パッケージのインストールでエラーになる場合の対処法

手順2の「apt install -y python3-venv」を実行した際、環境によっては以下の2つのトラブル（エラー）が発生して進行が止まることがあります。その場合は、一時的に管理者（root）権限に切り替えて以下の手順で強制解決してください。

ケースA：国やサーバー側の都合で「404 Not Found」エラーが出る場合

Debian公式サーバーのパッケージ名簿が更新された直後などに、ダウンロードURLの不一致（404）が発生します。この場合は、古い名簿キャッシュを物理的に一度リセットして再取得するのが確実です。

ケースB：画面に「apt-listchanges: News / (press q to quit)」と表示されて止まる場合

Debianの安全機能により、重要なセキュリティ変更点のニュースが表示されています。故障ではありませんので、キーボードの q キーを1回押すことで、自動的にインストールの続きが再開されます。

# 1. 一時的に管理者（root）ユーザーに切り替える（rootのパスワードを入力） $ su - パスワード: #   # 2. 【ケースA対策】古い名簿キャッシュを物理削除して、まっさらに更新する # rm -rf /var/lib/apt/lists/* # apt update   # 3. パッケージのインストールを再実行する # ※【ケースB】のニュース画面が出たら、キーボードの「q」を押して進めてください # apt install -y python3-venv   # 4. 不要になった古いシステムファイルを掃除する # apt autoremove -y   # 5. 管理者画面を終了し、元の一般ユーザーに戻る # exit logout $ ← 元の画面に戻れば成功！
