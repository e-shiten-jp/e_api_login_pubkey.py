立花証券ｅ支店API PKI認証 導入・設定マニュアルv4r9

（PKCS#12暗号化・自動運用設定編）

ダウンロードしたAPIプログラムをLinux環境に導入し、認証IDや秘密鍵を安全に暗号化（PKCS#12相当のセキュア保存）した上で、systemd を用いて自動実行するための設定手順書です。

動作確認環境は、API v4r9 / Python 3.13.5 / debian13 です。

1. ディレクトリ構成の作成と権限設定

安全な運用のために、実行用ディレクトリと鍵を格納する隠しディレクトリを作成し、所有者以外のすべてのユーザーからのアクセスを遮断（権限700）します。

推奨ディレクトリ構成

~/e_api/                     ← API実行基盤（権限: 700 / 所有者のみアクセス可）
├── .pki/                    ← 鍵・暗号化データ格納（権限: 700）
│   ├── file_pwd2.txt        ← 第2パスワード保存ファイル（手動作成）
│   └── file_login_response.txt ← ログイン応答出力先（自動生成）
├── .venv/                   ← Python仮想環境（自動生成されます）
├── e_api_encode_pkcs12.py   ← 暗号化ユーティリティ
├── e_api_login_pkcs12.py    ← ログイン実行スクリプト
└── file_url_info.txt        ← API接続情報ファイル（手動作成）

設定コマンド

ターミナルを開き、以下のコマンドを順番に実行してディレクトリを作成・保護してください。

# [1] ディレクトリの作成
$ mkdir -p ~/e_api/.pki

# [2] ディレクトリの権限（パーミッション）制限
# 親ディレクトリを含め、所有者のみに全権限を付与し、他ユーザーを完全に締め出します
$ chmod 700 ~/e_api
$ chmod 700 ~/e_api/.pki

2. Python仮想環境の準備

システムのPython環境に影響を与えず、プログラムに必要な依存ライブラリ（通信・暗号化モジュール等）を独立してインストールするため、仮想環境を作成します。

# [1] e_api ディレクトリに移動
$ cd ~/e_api

# [2] 仮想環境（.venv）の作成
$ python3 -m venv .venv

# [3] 仮想環境の有効化（アクティベート）
$ source .venv/bin/activate

# [4] 必須パッケージのインストール
# ※ プログラムで使用する通信・暗号化用の外部ライブラリを一括導入します
# ※ Cryptodome（末尾にdomeがつく形式）を利用するため、pycryptodomex を指定します
$ pip install --upgrade pip
$ pip install urllib3 cryptography pycryptodomex

# （確認）指定した3つのライブラリが正しくインストールされたか確認
$ pip list

※以降の作業でターミナルを一度閉じた場合は、再度 cd ~/e_api && source .venv/bin/activate を実行して仮想環境に入ってください。

3. 準備するファイルの配置と初期設定

3.1. サイトからダウンロードしたファイルの配置

API運用サイトから取得した以下の4つのファイルを、先ほど作成した ~/e_api/.pki/ ディレクトリ直下に配置（コピー）してください。

  • e_api_authid.txt
  • e_api_private_key.pem
  • e_api_private_key.der
  • e_api_public_key.pem

3.2. プログラムファイルの配置

配布された以下の2つのPythonスクリプトを、~/e_api/ ディレクトリ直下に配置してください。

  • e_api_encode_pkcs12.py
  • e_api_login_pkcs12.py

3.3. API接続情報ファイル（手動作成）

接続先や暗証番号を記録するファイルを ~/e_api/file_url_info.txt として新規作成します。

$ nano ~/e_api/file_url_info.txt

環境（本番/デモ）に合わせて以下の内容を記述し、保存してください。文字コードはUTF-8で保存します（プログラム側でBOMコードは自動排除されます）。

{
    "sUrl": "https://demo-kabuka.e-shiten.jp/e_api_v4r9/",
    "sJsonOfmt": "5"
}

3.4. 第2パスワード保存ファイル（手動作成）

注文、訂正、取り消しの処理で個別に利用する第2パスワードファイル（ログイン処理では未使用）を ~/e_api/.pki/file_pwd2.txt として新規作成します。

$ nano ~/e_api/.pki/file_pwd2.txt

⚠️ 記述時の重要な注意点

このファイルには第2パスワードを保存します。その際、末尾に改行コードや余分な制御文字が混入しないよう注意して記述してください。

第2パスワードは、注文、訂正、取消で使います。慎重に取り扱ってください。また短い期間で変更してお使いください。

また、ログイン後に同じディレクトリに出来るログイン応答出力（file_login_response.txt）も同様に慎重に扱ってください。

3.5. 配置ファイルの権限変更

万が一に備え、配下のすべてのファイルも所有者のみ読み書き可能（600）に制限します。

$ chmod 600 ~/e_api/.pki/*
$ chmod 600 ~/e_api/file_url_info.txt
$ chmod 600 ~/e_api/*.py

4. 初回限定：鍵情報の暗号化と原本の削除

認証IDと秘密鍵、第2暗証番号を一つにまとめ、強固に暗号化した設定ファイル（secure_config.enc）を生成します。

4.1. 暗号化プログラムの実行

$ cd ~/e_api
$ python3 e_api_encode_pkcs12.py

4.2. 出力結果の記録（最重要）

実行が成功すると、画面に以下のようなメッセージが表示されます。

============================================================
【重要】暗号化ファイルの作成に成功しました。
作成されたファイル: /home/[実行ユーザー名]/e_api/.pki/secure_config.enc
------------------------------------------------------------
以下の文字列を、自動実行環境の「環境変数 (API_DECRYPT_KEY)」に設定してください。
この鍵はファイルとしてディスクに保存せず、環境変数としてメモリから注入します。

API_DECRYPT_KEY=gAAAAABm...（ランダムな長い文字列）...=

上の行の内容は /etc/systemd/system/e_api_login.service ファイルに設定してください。
============================================================

⚠️ 注意

画面に表示された API_DECRYPT_KEY=〜 の文字列は、次のステップで systemd の設定に使用します。必ず一時的にメモ帳などへ控えておいてください。

4.3. 生ファイルの安全な削除

暗号化ファイル（secure_config.enc）が正常に生成されたため、サーバー上に残った暗号化されていない原本ファイルは安全のために速やかに削除します。

$ rm ~/e_api/.pki/e_api_authid.txt
$ rm ~/e_api/.pki/e_api_private_key.pem
$ rm ~/e_api/.pki/e_api_private_key.der

5. systemdによるサービス化（基盤設定）

OS管理機能（systemd）にプログラムを登録することで、環境変数を安全に隠蔽しながら実行できる環境を整えます。

5.1. サービス定義ファイルの配置

配布された e_api_login.service を、システムの配置ディレクトリにコピーします（要管理者権限）。

$ sudo cp ~/e_api/e_api_login.service /etc/systemd/system/e_api_login.service

5.2. サービスファイルの編集

コピーしたファイルを編集し、実行ユーザー名や環境変数を各自の環境に書き換えます。例ではエディターnanoを使っていますが、ご自身の使い慣れたエディターで編集してください。

$ sudo nano /etc/systemd/system/e_api_login.service

以下の項目を、お客様の環境に合わせて修正してください。

  • User= : 実行するLinuxのユーザー名（例: [実行ユーザー名]）
  • WorkingDirectory= : ディレクトリのフルパス（例: /home/[実行ユーザー名]/e_api/）
  • Environment= : 手順4の4.2.でメモした API_DECRYPT_KEY の値を貼り付けます。
  • ExecStart= : 仮想環境内のPythonパスと実行スクリプトのフルパス

【設定例】

[Unit]
Description=My Python Crypto API Service
After=network.target

[Service]
Type=simple
User=[実行ユーザー名]
WorkingDirectory=/home/[実行ユーザー名]/e_api/

# 手順4で生成されたキーをここに設定
Environment="API_DECRYPT_KEY=gAAAAABm...（あなたのキー）..."

# 必ず仮想環境(.venv)下のpython3を指定してください
ExecStart=/home/[実行ユーザー名]/e_api/.venv/bin/python3 /home/[実行ユーザー名]/e_api/e_api_login_pkcs12.py

[Install]
WantedBy=multi-user.target

5.3. systemdの設定反映

設定ファイルを保存したら、以下のコマンドでシステムに設定をリロードさせます。

$ sudo systemctl daemon-reload

6. 運用・実行方法と障害対応

セットアップ完了後の、手動での実行方法、障害発生時のセーフティ仕様、およびタイマー（定時自動実行）の設定手順です。

6.1. 手動実行と状態確認の方法

プログラムをテスト起動したい場合、また実行したプログラムの処理状態や成否ログを確認したい場合は、以下のシステム管理コマンドを利用します。

■ サービスの手動起動（実行リクエスト）

$ sudo systemctl start e_api_login.service

■ サービスの実行ステータス・ログ確認

$ sudo systemctl status e_api_login.service

※ログにエラーが出ておらず、隠しディレクトリ内に ~/e_api/.pki/file_login_response.txt が正しく生成・復号されていれば、プログラムは正常に動作しています。

6.2. 通信タイムアウトと自動リトライ（運用上の重要仕様）

証券会社側のサーバーは、夜間や週末（土日祝日）のメンテナンス時間帯に通信が遮断されるケースがあります。本プログラムは運用時のハングアップを防ぐため、以下の通信セーフティロジックが自動作動します。

  • タイムアウト制限: サーバーからの応答が15秒間ない場合、通信を自動で切り上げます。
  • 自動リトライ: 通信失敗を検知すると、5秒間の待機挟んで最大3回まで自動的に再接続を試みます。
  • エラーハンドリング: すべてのリトライが失敗した場合は、きれいなエラーメッセージを標準エラー出力へ吐き出し、ステータス「1 (FAILURE)」で安全にプロセスを終了します。

詳細な生ログを直接確認したい場合は、以下のコマンドを実行してください。

$ journalctl -u e_api_login.service --no-pager -n 50

6.3. 指定時刻に自動実行する設定（タイマー設定）

毎日決まった時刻（例：毎朝 08:30）にAPIログインを自動で実行させるため、systemdのタイマー機能（.timer）を作成します。

手順 A: タイマー定義ファイルの新規作成

$ sudo nano /etc/systemd/system/e_api_login.timer

手順 B: タイマー内容の記述
開いたファイルに以下の内容を貼り付けて保存します。時間は運用の必要に応じて OnCalendar= の部分を書き換えてください。

[Unit]
Description=Run My Python Crypto API Service Daily

[Timer]
# 毎日 朝08時30分00秒 に実行する設定例
OnCalendar=*-*-* 08:30:00
# サーバー起動後、最初の実行を少し遅らせる安全弁（任意）
OnBootSec=2min
# 呼び出す対象のサービス（同名の.serviceファイルが自動指定されます）
Unit=e_api_login.service

[Install]
WantedBy=timers.target

手順 C: タイマーの有効化と自動起動登録
設定を保存後、システムに認識させ、OS起動時にもタイマーが自動で動き出すよう有効化します。

# 1. systemd 設定のリロード
$ sudo systemctl daemon-reload

# 2. タイマーの有効化と今すぐの起動
$ sudo systemctl enable --now e_api_login.timer

# 3. タイマーが正しくセットされているかの確認
$ sudo systemctl list-timers --all | grep e_api_login

※確認コマンドを実行し、次回実行予定時刻（NEXT）に指定した時間が表示されていれば、毎日の自動運用設定は完了です。

7. 最終環境確認チェックリスト

すべての設定が正常に完了しているか、以下のコマンドを実行して状態確認を行ってください。

  • 1. 認証関係ファイルの削除確認
    不要になった設定ファイル（txt, pem, der）が残っていないことを確認します。
    # 確認コマンドを実行 $ ls -l ~/e_api/e_api_authid.txt ~/e_api/e_api_private_key.* 2>/dev/null # 正しい表示結果（何も表示されなければ正常に削除されています） （何も表示されないこと）
  • 2. ディレクトリとファイルの権限（パーミッション）確認
    重要なファイルが、所有者のみ読み書き可能（600）に制限されているか確認します。
    # 確認コマンドを実行 $ ls -l ~/e_api/file_url_info.txt ~/e_api/*.py # 正しい表示結果（行頭が必ず「-rw-------」から始まっていること） -rw------- 1 user user 123 5月 31 12:00 /home/user/e_api/file_url_info.txt -rw------- 1 user user 4567 5月 31 12:00 /home/user/e_api/main.py
  • 3. Python仮想環境の有効化確認
    仮想環境に入った状態で、使用されるPythonのパスが正しいか確認します。
    # ディレクトリに移動して有効化 $ cd ~/e_api $ source .venv/bin/activate (venv) $ which python # 正しい表示結果（末尾が「/e_api/.venv/bin/python」になっていること） /home/user/e_api/.venv/bin/python # 仮想環境から抜ける (venv) $ deactivate $
  • 4. systemdサービス稼働状態の最終確認
    APIログインサービスがエラーなく、正常に起動・動作しているか確認します。
    # サービスの状態を確認 $ sudo systemctl status e_api_login.service # 正しい表示結果（「Active: active (running)」または「active (exited)」が表示されていること） e_api_login.service - Tachibana API Login Service Loaded: loaded (/etc/systemd/system/e_api_login.service; enabled; vendor preset: enabled) Active: active (running) since Sun 2026-05-31 12:00:00 JST; 5min ago Main PID: 1234 (python)
